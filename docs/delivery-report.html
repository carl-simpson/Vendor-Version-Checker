Task 1: Composer Vendor Checker — v1.1.0

Bug Fixes

getSupportedVendors() broken Method was incorrectly transforming vendor keys using explode('.', $domain)[0], causing vendor name mismatches and preventing packages from being matched. Removed the unnecessary transformation.
XTENTO picking M1 version Product pages list both Magento 1 and Magento 2 versions. Implemented version_select: highest strategy to correctly select the M2 version.

New Features

Packagist API fallback When website scraping fails (Cloudflare, no version on page), falls back to the Packagist API (repo.packagist.org/p2/) for latest stable version. Results labelled [via Packagist].
Packagist-only tracking Packages without vendor website patterns (Klaviyo, TaxJar, Yotpo, etc.) are checked directly via Packagist API.
Cloudflare detection HTTP 403 responses containing Cloudflare Managed Challenge markers detected and reported with clear messaging instead of raw error dumps.
UNAVAILABLE status New status for packages that can't be checked automatically. Previously showed misleading AHEAD_OF_VENDOR with empty version.
Version source tracking Each result now includes a source field indicating where the version was obtained from.

Coverage Expansion

Package coverage expanded from the original set to 35 tracked packages across the Dowlis store:

New website mappings:
amasty/gdpr-cookie, amasty/geoipredirect, amasty/module-gdpr, amasty/number, aheadworks/module-blog, mageplaza/module-smtp
New Packagist packages:
taxjar, webshopapps, klaviyo, yotpo (x2), paradoxlabs (x2), justuno, stripe, bsscommerce, mageworx (x2)

Private Composer Repository Support — v1.2.0

Implementation

Auto-detection Reads composer.json repository definitions and auth.json credentials. Matches packages to private repos via dist URL or notification URL from composer.lock.
Multi-format resolution Supports Composer V2 (p2/), V1 (p/), full packages.json, and Satis provider-includes with hash-based URL resolution.
Smart filtering Skips repo.magento.com, marketplace.magento.com, and internal Satis/GetJohn repos to avoid false matches against Magento core packages.
Auth expired handling When private repo queries fail (HTTP 403), reports "auth may be expired" with actionable guidance instead of generic errors.

Amasty Licensing Investigation

During the scan, all 14 Amasty packages returned HTTP 403 errors. Investigation revealed that Amasty deprecated all global Composer access keys on 1 January 2026, replacing them with project-based keys.

Action Required by Client
  1. Log in to Amasty customer portal
  2. Create a Project for the Dowlis Merchandise Store
  3. Assign all 14 purchased Amasty extensions to the project
  4. Generate new project-level Composer keys
  5. Update auth.json with new credentials
  6. Verify with composer update amasty/* --dry-run

Full migration steps documented in PHP-8.4-Compatibility-Report.md.

Approach & Methodology

How we acquired source code, scanned it, and built the Module Register.

1

Tool Selection

PHPCompatibility 9.3.5 SELECTED

Industry-standard SAST ruleset purpose-built for PHP version migration. Detects deprecated functions, removed extensions, reserved keyword conflicts, changed INI directives, and parameter behaviour changes across PHP versions.

PHP_CodeSniffer 3.13.5 + PHPCompatibility 9.3.5
SonarQube CONSIDERED

SonarQube’s PHP plugin covers general code quality (bugs, code smells, security hotspots) but does not offer the same depth of PHP version migration analysis. Its compatibility rules are limited to a small subset of deprecations. PHPCompatibility output can be imported into SonarQube if needed.

Scanning Process

Each vendor group was scanned separately, producing 28 JSON result files. Results were then attributed to individual Composer packages via file path matching.

phpcs --standard=PHPCompatibility --runtime-set testVersion 8.4 --report=json
Per-vendor isolation

28 separate scan runs — one per vendor group — to keep results traceable and debugging straightforward.

Per-package attribution

File path matching ensures an error in vendor/laminas/laminas-db/src/... is attributed to laminas/laminas-db, not to all 28 laminas packages.

What the statuses mean
PASS Zero errors and zero warnings. The package’s PHP code is fully compatible with PHP 8.4.
WARN Warnings only (no errors). Typically false positives (e.g. Knockout.js template syntax flagged as PHP ASP tags) or deprecated-but-still-functional constructs.
FAIL One or more errors detected. The code uses functions, extensions, or syntax that is removed or will cause runtime errors on PHP 8.4.

INVESTIGATION

Amasty Licensing — Blocker Discovered During Scan

This issue was discovered during source code acquisition and affects both the PHP 8.4 scan (Task 2) and ongoing Composer operations.

What Happened

HTTP 403 on All Requests

Every request to composer.amasty.com returns 403 Forbidden. Installs, updates, and version queries all fail.

Global Keys Deprecated

Amasty retired all global Composer access keys on 1 January 2026. The credentials in auth.json are expired global keys.

New System: Project Keys

Amasty now requires project-based keys — created per-project in the Amasty portal with specific extensions assigned to each.

UPDATE

Dashboard & Tooling Improvements — 13 February 2026

PHP 8.4 Table Layout Optimisation

The 9-column PHP 8.4 compatibility table was reworked with explicit column widths to prevent data-sparse columns (Errors, Warnings, Risk) from consuming space needed by Package and Recommendation. Numeric columns are now centre-aligned. The table scrolls horizontally below 960px rather than crushing content.

Files Popover & Per-Package CSV Export

The “affected files” dropdown was converted from an inline expanding panel to an absolute-positioned popover (620px wide) with shadow, z-index stacking above the sticky nav, and click-outside-to-close behaviour. Each popover now includes a header with file count and a CSV download button that exports the affected file paths for that specific package.

Contextual Descriptions & Visual Legends

Every data display section in the dashboard now includes a description explaining what the data represents, why it’s relevant, and how to interpret it. This covers: the store selector tabs, all-stores overview grid, section tabs, PHP 8.4 summary cards, risk breakdown cards, filter controls, single store summary cards, filter tabs, and both results tables. Visual legends with colour-coded status indicators were added to the cross-store update matrix and the single store results table.

Accessibility Improvements

Added the @alpinejs/collapse plugin for the Actions Required banner animation. All decorative SVG icons now carry aria-hidden="true". Both search inputs have explicit aria-label attributes. Vendor version status dots now use role="img" with aria-label so colour is not the sole status indicator.

Downloadable File Links

All inline references to CSV and JSON files across both delivery reports are now clickable download links. File references in deliverable tables link directly to the hosted file with download icons distinguishing downloads from navigation links.

Landing Page & Navigation

A dedicated landing page (index.html) was created as the entry point for all deliverables — linking to both delivery reports, the interactive dashboard, and all downloadable files (CSVs + JSON). All pages have consistent “Home” navigation back to the landing page.

Standalone Runner

Added run-check.php — a CLI script for running vendor version checks outside the Composer plugin context. Supports JSON, CSV, and table output formats with optional file output and result caching.

Version Control

The docs/ directory is now tracked in version control. All reports, scan data, and the interactive dashboard are committed alongside the module source code.

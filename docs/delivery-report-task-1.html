What Was Delivered

Bug Fixes (v0.1.1)

getSupportedVendors() broken Method was incorrectly transforming vendor keys using explode('.', $domain)[0], causing vendor name mismatches and preventing packages from being matched. Removed the unnecessary transformation.
XTENTO picking M1 version Product pages list both Magento 1 and Magento 2 versions. Implemented version_select: highest strategy to correctly select the M2 version.

New Features (v0.2.0)

Packagist API fallback When website scraping fails (Cloudflare, no version on page), falls back to the Packagist API (repo.packagist.org/p2/) for latest stable version. Results labelled [via Packagist].
Packagist-only tracking Packages without vendor website patterns (Klaviyo, TaxJar, Yotpo, etc.) checked directly via Packagist API.
Cloudflare detection HTTP 403 responses containing Cloudflare Managed Challenge markers detected and reported with clear messaging instead of raw error dumps.
UNAVAILABLE status New status for packages that can’t be checked automatically. Previously showed misleading AHEAD_OF_VENDOR with empty version.
Version source tracking Each result now includes a source field indicating where the version was obtained (Website, Packagist, or Private Repo).
Coverage expansion Package coverage expanded from 10 to 22 tracked packages. New website mappings: amasty/gdpr-cookie, amasty/geoipredirect, amasty/module-gdpr, amasty/number, aheadworks/module-blog, mageplaza/module-smtp. New Packagist: taxjar, webshopapps, klaviyo, yotpo (x2), paradoxlabs (x2), justuno, stripe, bsscommerce, mageworx (x2).

New Features (v0.3.0)

Private Composer repo support Auto-detects private repositories from composer.json and matches credentials from auth.json. Supports Composer V2 (p2/), V1 (p/), full packages.json, and Satis provider-includes with hash-based URL resolution.
Concurrent HTTP requests warmCache/cachedGet pattern using Guzzle async promises. All vendor URLs pre-fetched in parallel before version extraction. Reduces total runtime from ~30s to ~3s.
CI/CD exit codes Exit code 0 = all current, 1 = updates available, 2 = errors. Enables use in CI pipelines with composer vendor:check; echo $?.
--packages filter composer vendor:check --packages=amasty/promo,xtento/orderexport to check specific packages only.
Improved Cloudflare detection Hybrid approach: checks response headers (cf-ray, cf-mitigated + 403) first, falls back to body content markers.
Dead code removal Removed unused methods: checkMultiplePackages, getComposerVersion (shell_exec security risk), getMarketplaceVersion, getMarketplaceUrl, checkAllPackages, displayResults.
Auth expired handling When private repo queries fail (HTTP 403), reports “auth may be expired” with actionable guidance. Smart filtering skips repo.magento.com and internal repos.

Blocker Resolution

Amasty Bot Detection — Resolved via 3-tier fallback

The original blocker (Amasty website scraping blocked by Cloudflare Managed Challenge v3) was addressed by implementing a 3-tier version resolution strategy:

  1. Website scraping — attempted first, Cloudflare detected and reported clearly
  2. Private Composer repo — queries composer.amasty.com/enterprise/ directly (requires valid project-level keys)
  3. Packagist API — fallback for packages on public Packagist (Amasty is not on Packagist)

The remaining gap is that Amasty’s Composer repo returns 403 because the auth.json credentials are expired global keys (deprecated Jan 2026). Once the client provides new project-level keys, the private repo check will work and bypass the Cloudflare-protected website entirely.

Commit History

All commits pushed to origin/master on Bitbucket — getjohn/module-composer-vendor-checker.

Why
getSupportedVendors() was treating vendor keys as domain names and splitting on ., which caused mismatches — packages couldn't be matched to their vendor entries in the config. This blocked all version checks from returning results.
What Changed
Removed the incorrect explode('.', $domain)[0] transformation. VersionChecker already returns clean vendor keys, so no transformation is needed.
Why
XTENTO's version page lists both Magento 1 and Magento 2 versions — the scraper was picking the M1 version, returning incorrect data. Several vendor websites also block automated requests with Cloudflare challenges, causing silent failures with no fallback.
What Changed
Implemented version_select: highest strategy with case-sensitive pattern to select the correct M2 version via version_compare. Added Packagist API as a fallback source when website scraping fails (results labelled [via Packagist]). Added Cloudflare Managed Challenge detection with clear messaging. Updated User-Agent to standard Chrome to reduce bot detection.
Why
The store uses many third-party packages that don't have dedicated version pages (Klaviyo, TaxJar, Yotpo, Stripe, etc.) — these were not being tracked at all. Additionally, some vendors serve pages that load but don't expose a parsable version number, which was silently ignored.
What Changed
Expanded tracking from ~10 to 22 packages. Added Packagist-only tracking for TaxJar, WebShopApps, Klaviyo, Yotpo, ParadoxLabs, Justuno, Stripe, BSS Commerce, and MageWorx. Introduced UNAVAILABLE status for pages that load but expose no version. Added version source tracking in report output.
Why
Multiple rounds of significant changes had been made without changelog documentation. A clear change history is needed for auditability and for the client to understand what was delivered.
What Changed
Added CHANGELOG.md with detailed entries covering all changes since v0.1.3 — bug fixes, Packagist fallback, Cloudflare detection, expanded coverage, private repo support, concurrent HTTP, and exit codes.
Why
The Cisco store uses a private Satis repository (cisco.satis.getjohn.co.uk) for custom modules. Without private repo support, the checker had no way to resolve latest versions for these packages — they were entirely invisible to the tool.
What Changed
Auto-detect private repos from composer.json repository definitions. Authenticate using auth.json HTTP basic auth credentials. Query private Satis/Composer repos supporting V2 (p2/), V1 (p/), full packages.json, and Satis provider-includes with hash-based URL resolution. Rewrote VendorCheckCommand::execute() to resolve auth paths from the lock file directory.
Why
The initial string-based Cloudflare detection was unreliable — it only checked the response body, which could miss challenge pages that return different HTML structures. The codebase also contained dead methods from earlier iterations, including getComposerVersion() which used shell_exec (a security risk).
What Changed
Replaced string-based detection with a hybrid approach: header-based detection (cf-ray, cf-mitigated) with body marker fallback for reliable challenge page identification. Removed unused methods: checkMultiplePackages, getComposerVersion, getMarketplaceVersion, getMarketplaceUrl.
Why
With 29 packages to check, sequential HTTP requests made execution slow (>30 seconds). The tool also had no machine-readable exit codes, making it unsuitable for CI/CD pipelines. Developers had no way to check a single package without running the full suite.
What Changed
Implemented warmCache() / cachedGet() using Guzzle async promises for concurrent HTTP requests, significantly reducing execution time. Added exit codes: 0 = all current, 1 = updates available, 2 = errors — enabling CI/CD integration. Added --packages filter option to check specific packages only.
Why
The README still referenced Magento Marketplace comparison (no longer supported) and didn't document any of the new capabilities — private repos, multi-source resolution, Packagist-only tracking, or the new CLI options. Anyone picking up the project would have no guide to how it works.
What Changed
Complete README rewrite covering three version source types (website scraping, private Composer repos, Packagist API). Added private repo auto-detection docs, supported vendor status table, installation and usage instructions, architecture overview, and troubleshooting section. Removed outdated Marketplace references.

What Was Delivered

getSupportedVendors() broken

Method was incorrectly transforming vendor keys using explode('.', $domain)[0], causing vendor name mismatches and preventing packages from being matched. Removed the unnecessary transformation.

XTENTO picking M1 version

Product pages list both Magento 1 and Magento 2 versions. Implemented version_select: highest strategy to correctly select the M2 version.

Packagist API fallback

When website scraping fails (Cloudflare, no version on page), falls back to the Packagist API (repo.packagist.org/p2/) for latest stable version. Results labelled [via Packagist].

Packagist-only tracking

Packages without vendor website patterns (Klaviyo, TaxJar, Yotpo, etc.) checked directly via Packagist API.

Cloudflare detection

HTTP 403 responses containing Cloudflare Managed Challenge markers detected and reported with clear messaging instead of raw error dumps.

UNAVAILABLE status

New status for packages that can’t be checked automatically. Previously showed misleading AHEAD_OF_VENDOR with empty version.

Version source tracking

Each result now includes a source field indicating where the version was obtained (Website, Packagist, or Private Repo).

Coverage expansion

Package coverage expanded from 10 to 22 tracked packages. New website mappings: amasty/gdpr-cookie, amasty/geoipredirect, amasty/module-gdpr, amasty/number, aheadworks/module-blog, mageplaza/module-smtp. New Packagist: taxjar, webshopapps, klaviyo, yotpo (x2), paradoxlabs (x2), justuno, stripe, bsscommerce, mageworx (x2).

Private Composer repo support

Auto-detects private repositories from composer.json and matches credentials from auth.json. Supports Composer V2 (p2/), V1 (p/), full packages.json, and Satis provider-includes with hash-based URL resolution.

Concurrent HTTP requests

warmCache/cachedGet pattern using Guzzle async promises. All vendor URLs pre-fetched in parallel before version extraction. Reduces total runtime from ~30s to ~3s.

CI/CD exit codes

Exit code 0 = all current, 1 = updates available, 2 = errors. Enables use in CI pipelines with composer vendor:check; echo $?.

--packages filter

composer vendor:check --packages=amasty/promo,xtento/orderexport to check specific packages only.

Improved Cloudflare detection

Hybrid approach: checks response headers (cf-ray, cf-mitigated + 403) first, falls back to body content markers.

Dead code removal

Removed unused methods: checkMultiplePackages, getComposerVersion (shell_exec security risk), getMarketplaceVersion, getMarketplaceUrl, checkAllPackages, displayResults.

Auth expired handling

When private repo queries fail (HTTP 403), reports “auth may be expired” with actionable guidance. Smart filtering skips repo.magento.com and internal repos.

Blocker Resolution

Amasty Bot Detection
Cloudflare Managed Challenge v3 — resolved via 3-tier fallback

The original blocker (Amasty website scraping blocked by Cloudflare) was addressed by implementing a 3-tier version resolution strategy:

1 Website Scraping

Attempted first. Cloudflare detected and reported clearly instead of failing silently.

Blocked by Cloudflare
2 Private Composer Repo

Queries composer.amasty.com/enterprise/ directly with auth credentials.

Needs new keys
3 Packagist API

Fallback for packages on public Packagist. Amasty is not published on Packagist.

Not available

Remaining gap: Amasty’s Composer repo returns 403 because the auth.json credentials are expired global keys (deprecated Jan 2026). Once the client provides new project-level keys, the private repo check will bypass Cloudflare entirely.
